Slide 08 / Solution — Before vs After

From 30 minutes to 30 seconds.

Before
Manual
  1. Open laptop, open syllabus PDF.
  2. Open Google Calendar in another tab.
  3. For each course: copy title, dates, time, location.
  4. Set up RRULE manually (every Monday & Wednesday at…).
  5. Repeat 5–7 times per semester. Hope nothing was wrong.
Output: 5–30 events in 30–90 minutes, often with errors.
After
Working today
  1. Open Telegram, send photo / voice / text to @CalenMindBot.
  2. Backend routes to the right model: Sonnet for vision, Haiku for text, Whisper→Haiku for voice.
  3. Pydantic-validated ParsedEventSchema objects produced.
  4. Events created in DB and pushed to Google Calendar with RRULE.
  5. Bot replies with formatted confirmation in <60s.
Output: same 5–30 events in <60 seconds, >90% correct.
Input: photo / voice / text
Processing: hybrid AI router
State change: Calendar event with RRULE
Human-in-the-loop: confirm before sync

Slide 11 / Product Scope from PRD

What ships in the MVP — and what does not.

Source of truth: prd.json + docs/spec.md.

Core Features (P0)
  • F001   Google OAuth 2.0 — secure calendar.events access.
  • F002   Multimodal input — photo, voice, text.
  • F003   LLM entity extraction — title, time, location, RRULE.
  • F004   Idempotent Google Calendar sync.
  • F006   Telegram-native interface (aiogram 3 widgets).
Differentiating Scenarios
  • SC002   Photo syllabus → events.
  • SC003   Voice note → event.
  • SC005   Grid schedule parsing — the wedge competitors miss.
  • SC004   Freemium gate at 10 syncs / month.
  • SC001   Onboarding via Telegram + OAuth.
Non-Functional
  • Async stack: SQLAlchemy + asyncpg + aiogram 3 (no blocking I/O).
  • Fernet AES-128-CBC encryption for OAuth tokens.
  • Pydantic v2 schemas at every boundary.
  • Bot has zero DB access; backend is independently scalable.
MVP Boundary
  • Telegram + Google Calendar only.
  • Russian + English language support.
  • Outlook, Apple Calendar, Chrome extension — later.
  • Payment integration handled by Telegram Payments / Stripe link (post-MVP).
Slide 12 / Architecture & Defensibility

A small system, but a serious one.

Client Layer
aiogram 3 Telegram bot, widget pattern (Trigger → Code → Answer). Zero install for the user.
Backend
FastAPI, layered: API → Service → Repository → Model. Pydantic v2 contracts at every seam.
Data
PostgreSQL (users, events, oauth_tokens). Tokens encrypted with Fernet AES-128-CBC.
Integrations
Google Calendar API, Anthropic (Claude Sonnet 4.6 + Haiku 4.5), Groq (Whisper-large-v3).
Scalability
Bot and backend deploy independently. Async throughout. Stateless API behind a managed Postgres.
Technical Moat
BDD-locked data contract on AI output (catches hallucinations before any DB write). Domain-tuned grid prompt for SC005.
Slide 13 / AI Advantage

Three models, one router, one cost line.

Photo → Sonnet 4.6
~$0.015
Highest table/syllabus vision accuracy. Auto-detects JPEG/PNG/WEBP. One automatic retry on parse failure.
Text → Haiku 4.5
~$0.001
Cost-efficient extraction for clean strings; supports Russian and English.
Voice → Whisper + Haiku
~$0.002
Groq Whisper for sub-second STT, Haiku for entity extraction. Avoids OpenAI quota limits.
Quality Loop
  • BDD scenarios (SC002, SC003, SC005) act as a regression suite for prompt + model upgrades.
  • Pydantic validation rejects malformed AI output before any DB write.
  • Markdown-fence stripping + corrective retry handles model drift.
Data Flywheel (planned)
  • User-corrected events become labeled training data for grid extraction.
  • Confidence + retry signals power an internal eval harness.
  • Each prompt rev measured against the same fixture set in CI.
